- **Scope:** habit add/check-in flow work — medicine/nutrition/physical add forms, check-in rerouting (emotion explore, symptoms move), medicine meal-slot status, nutrition preset persistence + migration script, habitService/schema/type changes.

## Verdict

No crash-level bugs survived verification. Several initially suspicious candidates were **refuted** by reading the actual guards:

- The locked-preset name field is enforced server-side (`resolveNutritionName` re-derives the name), so a devtools-edited readOnly input cannot persist a wrong name.
- The checklist tap handler is gated by `hasDetailedCheckin` (checklist/page.tsx:303), which matches exactly the categories `handleEntryTap` routes — no activity falls through to a dead tap.
- Activity deletion is a soft archive (`archiveActivity`), and `findHabitActivityById` does not filter archived docs, so the new "Activity not found" throw in `saveMedicineCheckin` is unreachable through the app.
- The new `pending`/`partial` medicine check-in status is the intended feature of `ba557f4`, not a regression.

What remains: one silent-API-behavior issue, two real performance problems in `habitService`, and a cluster of duplication/dead-code cleanups.

## Findings (most severe first)

### 1. `updateActivity` silently discards a rename for preset-backed nutrition activities
`frontend/lib/services/habitService.ts:343`

`resolveNutritionName` returns the preset label whenever `nutritionPreset` is set, ignoring `patch.name` — and the API reports success.

> **Failure:** `PATCH {name: "my custom name"}` on a nutrition activity with `nutrition_preset` set → response 200, but the stored name remains the preset label. The only way to rename is to also clear `nutritionPreset`, which no client currently knows to do.

**Recommendation:** decide whether preset names are immutable by design — if so, return a validation error instead of silently ignoring the rename.

### 2. N+1 sequential DB round-trips in habitService
`frontend/lib/services/habitService.ts:265` (also ~594, ~234)

`getTodayEntries` awaits `ensureOccurrence` once per activity in a loop; `getMonthlySummary` queries occurrences per activity sequentially; `buildGridRows` issues one query per activity via `Promise.all`.

> **Failure:** a user with 20 habits pays 1+20 Mongo round-trips to render the checklist; at ~150 ms/query against Atlas that is ~3 s instead of ~300 ms with a batched `find({activity_id: {$in: ids}})` repository method.

**Recommendation:** add one `$in`-batched repository method; it fixes all three call sites. The checklist is the app's hottest path.

### 3. O(activities × days) linear scans in `getMonthlySummary`
`frontend/lib/services/habitService.ts:611`

`occurrences.find()` runs inside nested activity×day loops instead of pre-indexing into a `Map`, the pattern `buildGridRows` already uses at line 236.

> **Failure:** 15 activities × 31 days = 465 linear scans per summary request; cost grows quadratically with occurrence count while the Map pattern 30 lines earlier makes each lookup O(1).

### 4. Orphaned legacy check-in routes
`frontend/app/(authed)/habit/checkin/emotion/page.tsx:1`, `frontend/app/(authed)/habit/checkin/symptom/page.tsx:1`

The checklist now routes to `/habit/checkin/physical/emotion/explore` and `/habit/checkin/physical/symptoms`, and the old literals were removed from the `Route` union in `types/navigation.ts` — but the old page files remain.

> **Failure:** two unreachable pages keep compiling and silently drift from the live forms; a future grep-driven edit (or a restored old link) resurrects a stale check-in form that no longer matches the data model.

### 5. Nutrition flow isolated by copying the medicine form wholesale
`frontend/app/(authed)/habit/add/nutrition/NutritionFormClient.tsx:40`

`formReducer`, `WEEKDAY_LABELS` (also in `physical/form/page.tsx:74`), and the frequency→Thai-label ternaries are near-identical triplicates across medicine, nutrition, and physical forms instead of one parameterized form component.

> **Failure:** any change to frequency auto-select semantics, weekday labels, or validation must be patched in three files; a missed copy gives medicine and nutrition forms divergent behavior for the same schedule input.

### 6. Preset-key normalization implemented three times
`frontend/app/(authed)/habit/add/medicine/page.tsx:26`

Trim/lowercase/dash→underscore + membership check exists as `resolveNutritionPreset` here, `toPresetKey` in `nutrition/[preset]/page.tsx:7`, and `isNutritionPresetKey` in `habitService.ts:47`.

> **Failure:** if the preset key format changes, the three copies drift and a URL accepted by one page is rejected by another, producing redirect loops or unrecognized-preset fallbacks.

### 7. Redundant client-side display-name resolution
`frontend/app/(authed)/habit/checklist/page.tsx:78`

`getActivityDisplayName` re-resolves the nutrition preset label, but `habitService.rowToActivity` already canonicalizes `activity.name` via `resolveNutritionName` before it reaches the client. The helper is also applied inconsistently (rows yes, delete/skip aria-labels at lines 334/346 no) — harmless today only because the server already resolved the name.

> **Failure:** two name-resolution layers invite drift: if preset label resolution changes server-side, the client helper shadows it; the inconsistent application misleads readers into thinking aria-labels show a different name.

### 8. Inconsistent encoding of the `from` query param
`frontend/app/(authed)/habit/add/nutrition/page.tsx:29` (also `medicine/page.tsx` ~228/237)

The nutrition back link interpolates `from` unencoded while the preset links on the same page use `encodedFrom`; medicine's `handleCancel`/`handleDiscard` also push unencoded `from`.

> **Failure:** today `from` is always a safe hardcoded path, but the first caller to pass a `from` containing `&` or `?` (e.g. `/habit/checklist?filter=daily`) gets a truncated redirect target from these three spots while the encoded links keep working.

### 9. Dead code in the medicine form
`frontend/app/(authed)/habit/add/medicine/page.tsx:242`

The `source` search param is read at line 128 and consumed only by `void source;`; the `CLEAR_ERRORS` reducer action is never dispatched anywhere in the file.

> **Failure:** readers assume `source` influences navigation and `CLEAR_ERRORS` is part of the live flow; both inflate the reducer/union and invite half-informed future edits.

### 10. Meal-slot Thai labels duplicated
`frontend/app/(authed)/habit/checkin/medicine/page.tsx:55`

`SLOT_LABEL` (breakfast→เช้า, lunch→กลางวัน, dinner→เย็น, before-bed→ก่อนนอน) duplicates the labels already built inline in `habitService.ts` `getSubline` (~149).

> **Failure:** adding a meal slot or fixing a translation requires touching both maps; a missed one shows different slot names on the checklist subline versus the check-in form.

## Suggested priorities

1. **Finding 1** — decide on preset-name immutability and surface an error instead of silent discard.
2. **Finding 2** — batch occurrence queries with a single `$in` repository method (also resolves finding 3's call site naturally while you're in the file).
3. Findings 4 and 9 — quick deletions, near-zero risk.
4. Findings 5–7, 10 — fold into the next refactor pass over the habit forms.
